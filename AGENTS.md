# Formula 1 AI Analyst

## Project Overview

This repository contains a production-style portfolio project showcasing Snowflake Cortex Analyst.

The application allows users to ask natural language questions about Formula 1 historical data. Cortex Analyst converts those questions into SQL against a curated semantic model built on Snowflake.

This project is intended for demonstration purposes in an Upwork portfolio. The primary goal is to showcase Snowflake AI capabilities, dimensional modeling, and clean software engineering practices.

---

# Project Objectives

The finished application should demonstrate:

- Snowflake as the analytical database
- Cortex Analyst for Text-to-SQL
- Semantic Models
- Streamlit frontend
- Clean SQL transformations
- Production-style project organization

The project should prioritize clarity, maintainability, and code quality over implementing every possible Formula 1 statistic.

# Engineering Principles

Follow these principles throughout the project.

## Keep It Simple

Prefer simple solutions.

Avoid unnecessary abstractions.

Do not introduce frameworks unless they provide significant value.

---

## Production Quality

Although this is a portfolio project, all code should resemble production code.

Use:

- logging
- type hints
- modular functions
- descriptive names
- environment variables
- docstrings

Avoid:

- hardcoded credentials
- duplicated logic
- excessively large files
- magic numbers
- unnecessary comments

---

## Incremental Development

Implement the project in small milestones.

Do not build multiple major features in one iteration.

Each milestone should leave the project in a working state.

# Python Standards

Target Python 3.11+

Follow PEP 8.

Prefer:

- pathlib
- dataclasses where appropriate
- pandas
- logging
- typing

Every public function should include a docstring.

---

# SQL Standards

Snowflake SQL only.

Analytics models should use:

- clear aliases
- consistent naming
- uppercase SQL keywords
- snake_case identifiers

Avoid deeply nested queries when possible.

Prefer readable CTEs.

---

# Snowflake Standards

Never hardcode:

- account
- username
- password
- warehouse
- database
- schema
- role

Use environment variables.

Use the Snowflake Python Connector unless another library is explicitly requested.

---

# Semantic Model

The semantic model should expose business concepts rather than raw database fields.

Examples:

Good:

- Driver
- Constructor
- Race
- Circuit
- Wins
- Podiums
- Championship Points

Avoid exposing technical column names directly whenever possible.

Include:

- descriptions
- synonyms
- verified queries
- measures
- dimensions
- time dimensions

---

# Streamlit

The UI should remain intentionally simple.

Required sections:

- title
- sidebar
- example prompts
- chat interface
- generated SQL
- result table

Optional:

- charts
- download results

Avoid unnecessary animations or visual clutter.

# Testing

When implementing new functionality:

- verify imports
- verify type hints
- verify no syntax errors

Prefer small unit tests where practical.

---

# Definition of Done

The project is complete when a user can:

1. Create Snowflake raw tables and analytics views.
2. Upload a semantic model.
3. Ask questions using natural language.
4. Receive SQL generated by Cortex Analyst.
5. Display query results in Streamlit.

---

# Agent Workflow

Before making significant changes:

1. Explain the implementation plan.
2. List the files that will be modified.
3. Keep changes as small as practical.
4. Preserve existing project structure unless a clear improvement exists.

If requirements are ambiguous, prefer asking for clarification over making assumptions.

When generating code:

- prioritize readability
- avoid premature optimization
- keep functions focused
- maintain consistent naming throughout the project

The objective is to create a portfolio-quality codebase that demonstrates professional software engineering practices.
